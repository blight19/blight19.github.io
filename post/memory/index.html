内存管理总结

Posted by 闫的Blog on Monday, June 4, 2018

虚拟内存

由于多进程运行，他们之间都需要使用物理内存，而需要让他们之间能够共用物理内存并减少内存碎皮的产生，所以需要使用虚拟内存。

虚拟内存有内存分段和内存分页两种方式。

内存分段

程序由若干逻辑分段组成，数据段，栈段，堆段，所以使用分段的形式将他们分开。

由段选择因子+段内偏移量构成，段选择因子相当于一个索引，里面保存的是段的基地址，界限，和特权等级等。

基地址加上段内偏移量得到物理内存地址。

存在问题：

  • 内存碎片(多进程归还内存的时候无法匹配其他进程的颗粒度)

  • 内存交换的效率低(多进程切换回频繁触发内存和硬盘的swap)

内存分页

分页是将虚拟内存和物理内存都切成固定尺寸的大小，linux是4kb，这4kb就叫做一个页。

虚拟内存映射到物理内存通过页表(页表也保存在内存中)和MMU(内存管理单元)转换。

当进程访问虚拟地址在页表中查不到的时候，会触发缺页异常，进入内核空间分配物理内存，更新进程页表，然后再返回用户空间。

但是内存分页分配内存最少分配一个页，会造成一些浪费。

如果内存空间不够，操作系统会将其他正在运行中的进程的最近没有被使用的页释放并写到硬盘上，这个行为叫做swap out.需要的时候再从硬盘加载到内存(swap in)。

页表一般分成多级，目的是为了节省页表的空间。

如果是二级页表，那么开始的时候只有一级的所有页表被创建，二级页表只有被使用的时候才被创建，而大多程序不会使用那么多的内存。

32位系统举例来说，整个4G的空间映射需要4MB的内存页表，64位会更大。而且每个进程都需要自己的页表。

64位的操作系统是四级页表

  • 全局页目录项PGD(page global directory)

  • 上层页目录项PUD(page uper ditectory)

  • 中层页目录项PUD(page middle ditectory)

  • 页表项PTE(page table entry)

多级页表的查询虽然空间占用少了，但是时间上来说肯定是变长了，所以引入了TLB(translation lookaside buffer)，这里是针对多级页表的缓存，通常称为页表缓存，TLB也在CPU内。

MMU会先访问TLB，没有才会去页表查询。

Linux中每个段的开始地址都是0。内存分段只被用于访问控制和内存的保护。

Linux内存管理主要采用页式内存管理，同时使用段机制。

段式内存管理先将逻辑地址映射成线性地址，然后再由页式内存管理将线性地址映射成物理地址。

在Linux中，虚拟地址空间又被分为内核空间和用户空间。32位一般是0-3G的是用户空间，3G到4G是内核空间。64位0-128T是用户空间，最后的128T是内核空间，中间是未定义空间。

用户进入用户态，只能访问用户空间内存

进入内核态以后才能访问内核空间的内存

malloc分配内存

分段：

  • 代码段 二进制可执行代码

  • 数据段 包括已初始化的静态常量和全局变量

  • BSS段 未初始化的静态变量和全局变量

  • 堆段 动态分配的内存，从低地址开始向上增长

  • 文件映射段 动态库，共享内存等

  • 栈段 局部变量和函数调用的上下文等。栈的大小是固定的，一般是8M，可修改

malloc其实是c库里面的函数，用于动态分配内存。有两种方式

  • 通过brk()系统调用从堆分配内存

  • 通过mmap()系统调用从文件映射分配内存

malloc从源码中定义了一个默认阈值

  • 如果分配内存小于128kb,通过brk申请堆段的内存

  • 如果大于128kb，通过mmap申请文件映射段的内存

注意，不同的glibc版本定义的阈值是不一样的

malloc分配的也是虚拟内存，如果仅进行了分配，没有进行访问的话，是不会映射到物理内存的，只有访问的时候，才会查询页表，触发缺页中断，建立映射关系。

可以通过/proc/{pid}/maps 查看一个进程的内存分布情况

malloc(1)分配1字节的空间，小于128kb,实际查看maps中,是分配了132kb的空间到堆内存，并多出来16个字节用于记录分配空间的大小，用于free使用。

在free的时候，如果使用brk分配在了堆段，那么内存是还在堆段的，再次申请的时候能够复用，不需要再次从操作系统申请。如果是使用mmap分配的大于128kb的，则会归还给操作系统。

为什么使用两种方式？

频繁大内存分配会导致调用系统调用次数过多，CPU消耗大。

如果大小内存都放在一起，会产生更多的内存碎片。

内存分配和回收

内存分配的时候如果没有空闲的物理内存，内核会回收部分内存，回收内存有两种方式

  • 后台内存回收(kswapd)

  • 直接内存回收(direct reclaim)

首先启动的是后台内存回收，如果后台回收内存不够，则会直接内存回收，直接内存回收是会阻塞程序运行的，会卡住，等回收完成后继续回到程序运行。如果还是不够，会触发OOM Killer，杀掉某些进程以释放内存。

内存回收主要有两类

  • 文件页(file back page)：内核缓存的磁盘数据(buffer)和内核缓存的文件数据(cache)。而其中有脏页(还没刷到磁盘的)，需要先写入磁盘才能释放。

  • 匿名页(anonymous page)：这部分内存没有实际的载体，是程序的一些堆栈数据等，需要使用swap机制写入到磁盘中。

文件页和匿名页的回收都是基于LRU算法。实际系统维护了active_list链表和inactive链表。

可以从/proc/meminfo中查看他们的大小

# grep表示只保留包含active的指标（忽略大小写）
# sort表示按照字母顺序排序
[root@xiaolin ~]# cat /proc/meminfo | grep -i active | sort
Active:           901456 kB
Active(anon):     227252 kB
Active(file):     674204 kB
Inactive:         226232 kB
Inactive(anon):    41948 kB
Inactive(file):   184284 kB

回收内存的时候kswapd后台回收如果不刷脏页的话，是不会阻塞进程的，而刷脏和使用直接内存回收的方式都是阻塞进程，并会使得系统符合变高的。

linux提供了一个参数用于修改在回收内存的时候使用匿名页(swap)还是回收文件页.这个参数就是/proc/sys/vm/swappiness，值为0-100，默认60，设置为0为更倾向文件页的回收。

调整kswapd的内核参数，控制异步回收内存。

使用

sar -B 1

命令可以观察系统直接回收和后台回收的指标

其中

  • pgscank/s表示后台回收线程每秒扫描的页

  • pgscand/s表示直接扫描的page个数

  • pgsteal/s 为上面两个的和

页低阈值（pages_low）可以通过内核选项 /proc/sys/vm/min_free_kbytes （该参数代表系统所保留空闲内存的最低限）来间接设置。

调整参数可以让触发异步回收内存更早，尽量避免触发直接回收内存。

NUMA架构下的内存回收策略

NUMA架构是一种CPU的架构，

NUMA架构下CPU被分为多个组，每组CPU可以有独立的资源，而当自己组的资源不足的时候，可以通过总线访问其他组的资源，虽然速度回慢。

 /proc/sys/vm/zone_reclaim_mode里面的参数控制了这个行为。

  • 0 （默认值）：在回收本地内存之前，在其他 Node 寻找空闲内存；
  • 1：只回收本地内存；
  • 2：只回收本地内存，在本地回收内存时，可以将文件页中的脏页写回硬盘，以回收内存。
  • 4：只回收本地内存，在本地回收内存时，可以用 swap 方式回收内存。

在使用 NUMA 架构的服务器，如果系统出现还有一半内存的时候，却发现系统频繁触发「直接内存回收」，导致了影响了系统性能，那么大概率是因zone_reclaim_mode 没有设置为 0 ，导致当本地内存不足的时候，只选择回收本地内存的方式，而不去使用其他 Node 的空闲内存。一般建议采用默认值0.

如何保护一个进程不被oom killer杀掉

在进行oom killer的时候，有两个因素会进行评分。

  • 进程使用的物理内存page数量

  • 每个进程的 OOM 校准值 oom_score_adj(/proc/[pid]/oom_score_adj)

计算方式

// points 代表打分的结果
// process_pages 代表进程已经使用的物理内存页面数
// oom_score_adj 代表 OOM 校准值
// totalpages 代表系统总的可用页面数
points = process_pages + oom_score_adj*totalpages/1000

oom_score_adj取值范围为-1000 到 1000，默认值为0.

如果不想某个进程被Kill，设置的越小越好（-1000）
